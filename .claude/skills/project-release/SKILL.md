# 子项目发布流程

在对话中说 **"发布子项目"** 或调用 `/project-release` 即可触发。

适用于在独立分支（`project/<name>`）上开发期间的版本发布。本流程与主发布流流程一致，包含代码审查、文档更新、复核等步骤。

## 流程

```
开发完成 → 说"发布子项目"
              │
              ▼
    ┌──────────────────────────┐
    │ ① 预检                   │ ← 分支、构建验证
    │   └ 可调用 /verify        │ ← 自动运行构建 + 验证产物
    └────────┬─────────────────┘
              │
              ▼
    ┌──────────────────────────┐
    │ ② 代码审查               │ ← git diff + 多维度审查
    │   ├ /code-review          │ ← 查 bug + 复用性 + 效率（必须）
    │   └ /simplify             │ ← 重构优化（可选）
    └────────┬─────────────────┘
              │
              ▼
    ┌──────────────────────────┐
    │ ③ 识别变更范围            │ ← 确定 commit 前缀 + 版本位 + 影响文档
    └────────┬─────────────────┘
              │
              ▼
    ┌──────────────────────────┐
    │ ④ 逐项更新 MD 文档        │ ← 按清单逐条更新，不得遗漏
    └────────┬─────────────────┘
              │
              ▼
    ┌──────────────────────────┐
    │ ⑤ 文档更新复核            │ ← 公示每份文档检查结果
    └────────┬─────────────────┘
              │
              ▼
    ┌──────────────────────────┐
    │ ⑥ 改动摘要               │ ← 变更清单 + 版本号，等待确认
    └────────┬─────────────────┘
              │  (用户确认)
              ▼
    ┌──────────────────────────┐
    │ ⑦ 自动执行               │ ← commit + tag + push
    └────────┬─────────────────┘
              │
              ▼
    ┌──────────────────────────┐
    │ ⑧ 合并到 main（可选）      │ ← 询问是否走主发布流
    └──────────────────────────┘
```

## 详细步骤

### ① 预检

| 检查项 | 命令 | 说明 |
|--------|------|------|
| 确认分支 | `git branch --show-current` | 必须是 `project/<name>` |
| 工作区干净 | `git status --porcelain` | 无未提交文件 |
| 构建验证 | 子项目构建命令 | 如 `npm run build` 必须通过 |

> 可选调用 `/verify` 自动执行构建并验证产物正确性。

### ② 代码审查

对当前改动做多维度检查：

| 步骤 | Skill | 说明 | 必选？ |
|------|-------|------|--------|
| 2a | `/code-review` | 检查正确性、复用性、效率 — 发现 bug 直接修复 | **必须** |
| 2b | `/simplify` | 重构优化 — 代码瘦身、消除重复、提升可读性 | 可选（改动量大时推荐） |
| 2c | **敏感信息检测** | `git diff HEAD` + 手动审查 — 扫描密码/Token/密钥/连接串/API Secret 等明文 | **必须** |

> **2c 检测规则**：逐行审查 git diff，不得出现明文密码、Token、API Key、连接字符串含密码、密钥等敏感信息。发现后立即移除，改为 `***` 或环境变量引用。
> 先跑 code-review 确保正确性，再视情况跑 simplify 做质量提升。三项的结果都直接应用到工作区。

### ③ 识别变更范围

**3a. 确定 commit 前缀和版本位：**

| 变更内容 | Commit 前缀 | 版本位 |
|---------|------------|--------|
| 新功能、新模块 | `feat` | `minor` |
| Bug 修复 | `fix` | `patch` |
| 文档更新 | `docs` | `patch` |
| 样式调整 | `style` | `patch` |
| 重构 | `refactor` | `patch` |
| 配置/依赖/CI | `chore` | `patch` |

**3b. 列出本次影响范围：**

分析 `git diff` 涉及的变更，向用户说明：
1. 本次改动了子项目的哪些模块
2. 是否影响子项目的文档（CHANGELOG / README / PLAN）
3. 是否影响根仓库文档（CLAUDE.md 等）

### ④ 逐项更新 MD 文档

**强制规则：必须按下方清单逐条检查，每检查完一项向用户反馈状态。**

| # | 文档 | 更新条件 | 本次？ | 检查内容 |
|---|------|---------|--------|---------|
| 1 | `CLAUDE.md`（根） | **每次必更** | 必更 | 更新当前工作状态、进度标记 |
| 2 | `{子项目}/CHANGELOG.md` | **每次必更** | 必更 | 新增版本条目 |
| 3 | `{子项目}/README.md` | 技术栈/结构/命令变动 | 检测 | 保持与实际一致 |
| 4 | `{子项目}/PLAN.md` | 方案设计变动 | 检测 | 与技术实现不一致时更新 |
| 5 | 根 `README.md` | 子项目表/目录结构变动 | 检测 | 子项目列表同步 |
| 6 | 根 `PLAN.md` | 子项目生态/路线图变动 | 检测 | 与项目全景一致 |

> **执行方式**：逐项读文件 → 判断是否需要更新 → 如需要则立即编辑 → 完成后打 ✅ 标记。
> **敏感信息红线**：所有受检文档中不得出现明文密码、Token、连接串含密码、API Secret 等。编辑时发现需立即移除（改为 `***` 或环境变量引用）。

### ⑤ 文档更新复核

**必须在进入步骤⑥之前执行。** 向用户输出复核报告：

```
📋 文档更新复核

CLAUDE.md                   ✅ 已更新 / ⏭ 无需更新
{子项目}/CHANGELOG.md       ✅ 已更新 / ⏭ 无需更新
{子项目}/README.md          ✅ 已更新 / ⏭ 无需更新
{子项目}/PLAN.md            ✅ 已更新 / ⏭ 无需更新
根 README.md                ✅ 已更新 / ⏭ 无需更新
根 PLAN.md                  ✅ 已更新 / ⏭ 无需更新
```

**如果任何必更项显示 ⏭，必须在进入步骤⑥之前修复。**

### ⑥ 改动摘要

```
子项目：{子项目}
版本变化：v旧版本 → v新版本（minor/patch）

文件变更清单
  ├── 新增：xxx
  ├── 修改：xxx
  └── 删除：xxx

Commit 信息预览
  └── release: {子项目} v{新版本}
```

等待用户回复"ok"或"确认"后继续。

### ⑦ 自动执行

```bash
# 选择性暂存子项目文件
git add {子项目}/

# 提交
git commit -m "release: {子项目} v{新版本}"

# 打子项目标签（与主仓库 tag 区分，如 monitor-v0.2.0）
git tag {子项目}-v{新版本}
git push && git push --tags
```

### ⑧ 合并到 main（可选）

询问是否合并到 main 并走主发布流：

```
当前 {子项目} v{新版本} 是否已准备好合并到 main？
  → 是 → 自动执行：
      git checkout main
      git merge project/{子项目}
      git branch -d project/{子项目}  # ⚠️ 仅当子项目不再需要开发时才删除（如废弃/归档）
      合并完成，请说"提交代码"走主发布流
  → 否 → 流程结束，继续开发
```

## 禁止事项

- ❌ 跳过步骤④直接进入摘要
- ❌ 改为"可以下个版本再更"（必须本版本更新）
- ❌ 用户说"ok"时跳过复核直接提交
- ❌ 遗漏 CHANGELOG.md 更新
- ❌ 摘要阶段才发现有未更新的文档
- ❌ 在代码或文档中明文写入密码、Token、API Key、连接字符串含密码、密钥等敏感信息
